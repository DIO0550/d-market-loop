# Task Loop Doc

ユーザーの要件をタスクファイルに分割して生成し、カンバン形式のタスクダッシュボード（`Task.md`）を生成する。

- タスクファイルフォーマットの詳細は `references/task-file-format.md` を参照。
- ダッシュボードフォーマットの詳細は `references/task-md-format.md` を参照。

## 前提条件

1. タスクディレクトリが存在すること — 設定の `tasksDir`（デフォルト: `tasks`）

前提条件が整っていない場合は `task-loop-setup` スキルの使用を案内する。

## 設定の読み込み

リポジトリルートの `task-loop-config.json` を読み込む。ファイルが存在しない場合は全てデフォルト値を使用する。

使用するフィールド:
- `tasksDir`（デフォルト: `tasks`）— タスクファイルの格納ディレクトリ
- `planFile`（デフォルト: `Task.md`）— ダッシュボードの出力先パス

## 生成フロー

以下のステップを順番に実行する。

### Step 1: タスクファイルの生成

ユーザーにタスクの入力方法を確認する:

**方法A: 直接入力**
ユーザーが実装したい内容を説明する。それを適切な粒度のタスクファイルに分割して生成する。

**方法B: 既存ドキュメントからの変換**
ユーザーが指定するドキュメント（計画書、設計書、issueリスト等）を読み込み、タスクファイルに分割して生成する。

**方法C: GitHub Issue からの変換**
`gh issue list` や `gh issue view` を使ってissueを読み込み、タスクファイルに変換する。

いずれの方法でも、`references/task-file-format.md` に従って `{tasksDir}/todo/` に生成する。

#### タスク分割ガイド

- **1タスク = 1PR = 1つの独立した変更**にする
- タスクは単独でコミット・PRできる粒度にする
- 依存関係がある場合はファイル名の連番で順序を表現する
  - 例: `001-create-model.md` → `002-add-api-endpoint.md` → `003-add-frontend-page.md`
- タスクが大きすぎる場合は更に分割する（目安: 1タスクで変更するファイルが10個以下）
- テストとコードは同じタスクに含める（テストだけの別タスクにしない）
- 設定変更や依存追加は、それを使う実装タスクと同じタスクに含める

### Step 2: プロジェクトコンテキストの収集

リポジトリを分析してプロジェクトの技術的コンテキストを収集する。

1. **技術スタックの自動検出**:
   - `package.json` → Node.js / npm の依存関係、フレームワーク、テストツール
   - `Cargo.toml` → Rust の依存関係
   - `go.mod` → Go のモジュール情報
   - `pyproject.toml` / `requirements.txt` → Python の依存関係
   - `Gemfile` → Ruby の依存関係
   - その他の設定ファイル（`tsconfig.json`、`next.config.*`、`vite.config.*` 等）
2. **アーキテクチャの把握**:
   - ディレクトリ構成を確認する
   - 主要なエントリポイントやレイヤー構成を把握する
3. **制約条件の確認**:
   - ユーザーに制約条件や特記事項がないか確認する
   - 既存のコーディング規約ファイル（`.eslintrc`、`.prettierrc`、`editorconfig` 等）があれば読み取る

### Step 3: タスク分析とカンバンセクション生成

タスクファイルを分析し、カンバンセクションを構成する。

1. `{tasksDir}/*.md` を全て読み込む
2. 各タスクファイルの frontmatter（`title`、`status`、`priority`）と本文を解析する
3. **タスクエントリの生成**:
   - 各タスクについて以下を記録する:
     - ID（ファイル名から `.md` を除いたもの）
     - タイトル（frontmatter の `title`、なければファイル名のハイフン区切り部分）
     - Priority（frontmatter の `priority`、デフォルト: `normal`）
     - File（タスクファイルへの相対パス）
4. **Blockers の推定**:
   - タスク内容を分析し、あるタスクが別のタスクの成果物を前提としているかを判定する
   - 依存先タスクの ID を Blockers フィールドに設定する
   - 依存がない場合は `none` とする
5. **Shared Context の抽出**:
   - タスク間で共有すべき情報（共通モジュール、命名規則、データ構造等）を特定する
   - Context セクションの Shared Context に記述する
6. **セクションへの振り分け**:
   - `status` が未設定または `pending` → **Todo** セクション
   - `status` が `in_progress` → **Processing** セクション
   - `status` が `completed` → **Done** セクション
   - `status` が `failed` または `skipped` → **Failed** セクション
   - 初回生成時は通常すべて Todo に配置される

### Step 4: リスク・注意点の分析

1. 依存チェーンが長いタスクを特定し、リスクを記述する
2. 外部サービス依存、マイグレーション、破壊的変更などのリスクを洗い出す
3. Step 2 で検出した規約ファイルの内容をまとめる
4. タスク内容からレビュー時に注目すべきポイントを導出する
5. 上記をすべて Context セクションの **Notes** にまとめる

### Step 5: Task.md ファイルの出力

1. `references/task-md-format.md` のフォーマットに従って Task.md を組み立てる
2. frontmatter を設定する:
   - `title`: プロジェクト名とタスク群の目的を組み合わせる
   - `createdAt`: 現在時刻（ISO 8601）
   - `updatedAt`: 現在時刻（ISO 8601）
   - `totalTasks`: タスクファイル数
3. `planFile`（デフォルト: `Task.md`）のパスに出力する
4. 既にファイルが存在する場合はユーザーに上書き確認する

### Step 6: 生成結果の確認

生成したダッシュボードの概要をユーザーに提示する。

出力例:
```
タスクダッシュボードを生成しました。

ファイル: Task.md

概要:
  - プロジェクト: ECサイト認証機能
  - タスク数: 5
  - Todo: 5 / Processing: 0 / Done: 0 / Failed: 0
  - 主な依存チェーン: 001 → 002 → 003 → 004 → 005

内容を確認し、必要に応じて編集してください。
```
