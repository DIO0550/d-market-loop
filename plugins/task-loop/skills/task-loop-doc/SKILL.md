# Task Loop Doc

タスクファイルとプロジェクト情報を分析し、ループ全体計画書（`task-loop-plan.md`）を生成する。

計画書フォーマットの詳細は `task-loop-run/references/loop-plan-format.md` を参照。

## 前提条件

1. タスクディレクトリが存在すること — 設定の `tasksDir`（デフォルト: `tasks`）
2. タスクディレクトリにタスクファイル（`.md`）が1つ以上あること

前提条件が整っていない場合は `task-loop-setup` スキルの使用を案内する。

## 設定の読み込み

リポジトリルートの `task-loop-config.json` を読み込む。ファイルが存在しない場合は全てデフォルト値を使用する。

使用するフィールド:
- `tasksDir`（デフォルト: `tasks`）— タスクファイルの格納ディレクトリ
- `planFile`（デフォルト: `task-loop-plan.md`）— 計画書の出力先パス

## 生成フロー

以下のステップを順番に実行する。

### Step 1: プロジェクトコンテキストの収集

リポジトリを分析してプロジェクトの技術的コンテキストを収集する。

1. **技術スタックの自動検出**:
   - `package.json` → Node.js / npm の依存関係、フレームワーク、テストツール
   - `Cargo.toml` → Rust の依存関係
   - `go.mod` → Go のモジュール情報
   - `pyproject.toml` / `requirements.txt` → Python の依存関係
   - `Gemfile` → Ruby の依存関係
   - その他の設定ファイル（`tsconfig.json`、`next.config.*`、`vite.config.*` 等）
2. **アーキテクチャの把握**:
   - ディレクトリ構成を確認する
   - 主要なエントリポイントやレイヤー構成を把握する
3. **制約条件の確認**:
   - ユーザーに制約条件や特記事項がないか確認する
   - 既存のコーディング規約ファイル（`.eslintrc`、`.prettierrc`、`editorconfig` 等）があれば読み取る

### Step 2: タスク分析

タスクファイルを分析して全体像を把握する。

1. `{tasksDir}/*.md` を全て読み込む
2. 各タスクファイルの frontmatter（`title`、`status`、`priority`）と本文を解析する
3. **Execution Order テーブル**を作成する:
   - ファイル名の昇順でタスクを並べる
   - 各タスクのタイトルと概要を記録する
4. **依存関係の推定**:
   - タスク内容を分析し、あるタスクが別のタスクの成果物を前提としているかを判定する
   - Dependency Graph をテキストベースで作成する
5. **Shared Context の抽出**:
   - タスク間で共有すべき情報（共通モジュール、命名規則、データ構造等）を特定する
   - 先行タスクの実装が後続タスクに与える影響を記述する

### Step 3: リスク・注意点の分析

1. **Known Risks**:
   - 依存チェーンが長いタスクを特定し、ブロッキングリスクを記述する
   - 外部サービス依存、マイグレーション、破壊的変更などのリスクを洗い出す
2. **Coding Conventions**:
   - Step 1 で検出した規約ファイルの内容をまとめる
   - プロジェクト固有のパターンやルールを記載する
3. **Review Focus Points**:
   - タスク内容からレビュー時に注目すべきポイントを導出する
   - セキュリティ、パフォーマンス、テストカバレッジ等の観点

### Step 4: 計画書ファイルの出力

1. `references/loop-plan-format.md` のフォーマットに従って計画書を組み立てる
2. frontmatter を設定する:
   - `title`: プロジェクト名とタスク群の目的を組み合わせる
   - `createdAt`: 現在時刻（ISO 8601）
   - `totalTasks`: タスクファイル数
   - `estimatedDuration`: 推定所要時間（任意）
3. `planFile`（デフォルト: `task-loop-plan.md`）のパスに出力する
4. 既にファイルが存在する場合はユーザーに上書き確認する

### Step 5: 生成結果の確認

生成した計画書の概要をユーザーに提示する。

出力例:
```
ループ全体計画書を生成しました。

ファイル: task-loop-plan.md

概要:
  - プロジェクト: ECサイト認証機能
  - タスク数: 3
  - 依存チェーン: 001 → 002 → 003（直列）
  - 主なリスク: DBマイグレーション失敗時の全タスクブロック

内容を確認し、必要に応じて編集してください。
```
