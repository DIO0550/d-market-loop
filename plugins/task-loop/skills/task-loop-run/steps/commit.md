# コミット

> ⚠️ **1 タスク = 1 コミット ではない。** 複数の論理的単位にまたがる変更を 1 コミットに
> まとめると後からの差分追跡・revert・レビューが困難になる。以下のルールに従って
> **必ず意味のある単位で分割する**。

## Step 1: 変更を論理的単位に分割する

まず `git status` と `git diff` で全変更を確認し、以下の基準で**分割単位を決める**。
「分割することを検討する」ではなく、**該当する基準にヒットしたら必ず分割する**。

### 必ず分割する基準（いずれかにヒットしたら分割）

1. **異なる機能・関心事にまたがる変更**
   例: 認証ロジック追加 + ログ出力整備 → 2 コミット
2. **異なるレイヤーにまたがる変更**
   例: DB マイグレーション + API エンドポイント + UI コンポーネント → 3 コミット
3. **新機能の追加とリファクタリングの同居**
   例: 新しい関数を追加するついでに既存コードを整理 → 2 コミット
4. **テストコードと実装コード以外の無関係な変更**
   例: 新機能の実装 + 無関係な typo 修正 → 2 コミット
   （実装とそれに対応するテストは同一コミットで OK）
5. **依存追加・設定変更と実装の同居**
   例: `package.json` への依存追加 + その依存を使う実装 → ケースバイケース。依存追加だけで
   動く状態にできるなら分割、そうでなければ同一コミット可
6. **3 ファイル以上にまたがる変更** → デフォルトで分割を検討する。単一の論理的変更で
   あることを説明できない場合は分割する

### 同一コミットに含めてよい基準

- 同じ関数・同じファイルに対する 1 つの論理的変更（実装 + それに直結するテスト）
- rename / move のように機械的に波及する変更
- 型定義の追加とその型を使う実装（分離すると途中状態でビルドが通らなくなる場合）

### アンチパターン（やってはいけない）

- ❌ タスクの全変更を `fix: レビュー指摘対応` のような漠然としたメッセージで 1 コミット
- ❌ 「関連しているから」という理由で 5 ファイル以上を 1 コミットにまとめる
- ❌ 無関係な整形・typo 修正を本筋の変更に混ぜる
- ❌ 新機能追加 + 既存コードのリファクタを 1 コミットに同居

### 分割例

変更ファイル:
- `src/auth/jwt.ts`（新規: JWT 発行ロジック）
- `src/auth/jwt.test.ts`（新規: 上記のテスト）
- `src/api/login.ts`（新規: ログインエンドポイント）
- `src/api/logout.ts`（新規: ログアウトエンドポイント）
- `src/middleware/logger.ts`（整形のみ）

→ 3 コミットに分割:
1. `feat: JWT発行ロジックを追加`（jwt.ts + jwt.test.ts）
2. `feat: login/logout APIエンドポイントを追加`（login.ts + logout.ts）
3. `chore: loggerミドルウェアの整形`（logger.ts）

## Step 2: コミット単位ごとにステージング → コミット

Step 1 で決めた単位ごとに以下を **順番に** 繰り返す。1 コミット作ったら次の単位に進む。

### Step 2-A: 対象ファイルだけをステージング

- `git add -A` は使用禁止
- このコミット単位に属するファイルのみを個別指定する
- `.env`、認証情報、ビルド成果物など不要ファイルが含まれていないことを `git status` で確認

```bash
git add {この単位のファイル1} {この単位のファイル2} ...
```

### Step 2-B: コミットメッセージを作成

- フォーマット: `{commitPrefix}: {この単位の変更内容}`
- `commitPrefix` はタスクファイルの frontmatter から取得（デフォルト `feat`）。
  ただしこの単位が feat ではない場合は適切なものを使う:
  - `feat`: 新機能
  - `fix`: バグ修正
  - `refactor`: 挙動を変えないリファクタリング
  - `test`: テストの追加・修正のみ
  - `chore`: 設定・雑務・依存更新・整形
  - `docs`: ドキュメントのみ
- タイトルをそのまま使うのではなく、**その単位で何を変えたか** を簡潔に記述する
- 漠然とした要約（「修正」「対応」「調整」）は使わない

### Step 2-C: コミット実行

```bash
git commit -m "{コミットメッセージ}"
```

### Step 2-D: 次の単位があれば Step 2-A に戻る

Step 1 で決めた全ての単位をコミットし終えたら本ステップ完了。

## 終了前チェックリスト

- [ ] `git status` で未ステージの変更が残っていない
- [ ] `git log --oneline` で各コミットが単一の論理的単位を表している
- [ ] 「レビュー指摘対応」「リファクタ」のような漠然としたメッセージが無い
- [ ] 3 ファイル以上のコミットがある場合、それが単一の論理的変更であることを説明できる
