# セルフレビュー

実装完了後、コミット前にサブエージェントで自己レビューを行い、品質を担保する。

1. 現在の変更差分を取得する:
   ```bash
   git diff
   ```
2. Agent ツールを使用してサブエージェントにレビューを依頼する。プロンプトには以下を含める:
   - 変更差分（`git diff` の出力）
   - タスクファイルの Requirements と Acceptance Criteria
   - Task.md の Constraints・Notes（読み込まれている場合）
   - レビュー観点の指示（下記テンプレートの「レビュー観点」セクション参照）
3. サブエージェントの指摘を分析する:
   - **問題あり**: 指摘内容に従って修正を行い、再度セルフレビューを実行する（最大2回まで）
   - **問題なし**: 次のステップ（`steps/commit.md`）へ進む
4. 2回修正しても指摘が解消しない場合は、残存する指摘をコミットメッセージの補足情報として記録し、次のステップへ進む（外部レビューに委ねる）

## サブエージェントプロンプトのテンプレート

```
以下の変更差分をレビューしてください。

## タスク要件
{タスクファイルの Requirements セクション}

## 受け入れ基準
{タスクファイルの Acceptance Criteria セクション}

## 制約・規約
{Task.md の Constraints・Notes、またはなければ省略}

## 変更差分
{git diff の出力}

## レビュー観点
1. 要件充足: Requirements・Acceptance Criteria を全て満たしているか
2. セキュリティ: インジェクション、XSS、認証不備、機密情報ハードコードがないか
3. エッジケース: 境界値、null/undefined、空配列・空文字列の考慮漏れ
4. 既存コードとの整合性: 命名規則、コードスタイル、アーキテクチャの一貫性
5. 不要な変更: スコープ外の変更、デバッグコード残留、不要なコメント

## 回答フォーマット
問題がない場合: 「LGTM」とだけ回答してください。
問題がある場合: 以下の形式で列挙してください。

- **[観点]** ファイル名:行番号付近 — 指摘内容と修正案
```

## 注意事項

- セルフレビューはあくまで事前チェックであり、外部レビューの代替ではない
- 軽微なスタイルの好みや主観的な改善提案は無視してよい（修正対象は明確なバグ・要件不足・セキュリティリスクに限定）
- サブエージェントの指摘が的外れと判断した場合はスキップしてよい
